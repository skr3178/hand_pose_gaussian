# Section 4-2 — Human Data: Learning from Human Hands

Reproduction notes for the RLDX-1 human-hand data pipeline.

## TL;DR

- **The §4-2 pipeline is NOT in the released repo.** RLWRLD shipped the *consumer*
  of the data (the VLA trainer, which expects LeRobot v2.1 datasets) but **not the
  producer** (the hand-track → 3DGS → retarget → sim-rollout pipeline).
- To run §4-2 you assemble ~4 separate open-source projects yourself, driven by
  multi-view video you record. Only **stage 4** overlaps with this repo.
- The model checkpoints are public and downloadable; the data pipeline is not.

## What the paper describes

> Record from the bare human hand and close the kinematic/morphological gap in
> software. Four stages → 200+ demonstrations/hour.

| Stage | Goal |
|---|---|
| 1 | Track the human hand and object |
| 2 | Reconstruct the workspace with 3D Gaussian Splatting |
| 3 | Retarget human hand motion onto the robot hand |
| 4 | Roll out in simulation to produce VLA training data |

## Inputs you must provide yourself (not downloadable)

> **Deviation in our reproduction:** the paper assumes multi-view capture, but we
> are building from **monocular RGB video** — see `goal.md` (which wins where the
> two documents disagree) for the monocular pipeline and its extra risks.

- Multi-view RGB(-D) video of a human hand performing the task.
- The robot hand's kinematic model (URDF) — e.g. ALLEX.
- (Stage 1 object track) a CAD/mesh of the manipulated object, if using a
  model-based object tracker.

## Stage-by-stage build (third-party substitutes)

> None of these are RLWRLD's exact stack — they are the standard open-source
> components that fill each role.

### Stage 1 — Track hand + object
- **Hand:** per-frame 3D hand mesh / 21 keypoints (MANO).
  - Download: **HaMeR** or **WiLoR** (pretrained, single-image → 3D hand).
  - Output: per-frame joint angles + wrist 6-DoF pose.
- **Object:** 6-DoF pose track.
  - Download: **FoundationPose** (needs object mesh) or **BundleSDF** (model-free).
- **Hardest part:** occlusion during contact (fingers + object occlude each other).
  Most of the effort goes here; bad tracking silently poisons everything downstream.

### Stage 2 — 3DGS workspace reconstruction
- Download: `graphdeco-inria/gaussian-splatting` (paper ref [22]), or **gsplat** /
  **Nerfstudio**.
- Run **COLMAP** first for camera poses.
- Note: vanilla 3DGS is **static-scene** → mask out the moving hand/object using
  the stage-1 tracks.

### Stage 3 — Retarget human → robot hand
- Download: **`dex-retargeting`** (from AnyTeleop) — purpose-built, multi-hand;
  supply the robot-hand URDF.
- Solves an optimization mapping human keypoints → robot joint angles, closing the
  morphological gap (finger count/length/workspace differ).
- **Hardest part:** infeasible targets — some human grasps have no valid robot
  equivalent. Add a feasibility/collision filter to drop bad frames.

### Stage 4 — Sim rollout → VLA data (only part overlapping this repo)
- Replay retargeted trajectories in a physics sim: **Isaac Lab** / **MuJoCo** /
  **RoboCasa** (already a submodule under `external_dependencies/`).
- Record rollouts, then export to **LeRobot v2.1** via `run_scripts/data/` so the
  RLDX-1 trainer can consume them.

## Major caveats / open questions

- **No force/tactile capture.** Bare-hand video records *kinematics only*, yet the
  paper's whole thesis (Physics Module) is contact forces + torque + tactile. The
  highest-volume data source is blind to the modalities argued to be essential.
- **"200+ demonstrations/hour"** conflates human-capture rate with sim-augmentation
  output; the real validity/yield rate of rollouts is unstated.
- **Sim2real gap.** Stage-4 data is simulation data; dexterous in-hand contact is the
  worst case for sim fidelity.

## Recommended order (stop at first failure)

1. **Confirm the model runs first** — download a checkpoint
   (`RLWRLD/RLDX-1-MT-ALLEX`, already pulled to `checkpoints/RLDX-1-MT-ALLEX/`),
   build the pixi/CUDA env (Python 3.10 + torch 2.7 + flash-attn — separate from
   `selfocc`), run a test inference.
2. **Prototype Stage 1 alone** on your own footage. If hand tracking under
   contact-occlusion isn't reliable, the rest is moot.
3. Only then build stages 2 → 3 → 4.
